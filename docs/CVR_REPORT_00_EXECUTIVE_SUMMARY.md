## 全体結論: CVRを最大化する最適プラン設計

5つの専門領域の分析結果を統合し、THE TOLLのFree→Pro CVRを最大化するための推奨プランを以下にまとめる。

## 2. 推奨Free/Pro機能境界線（最重要変更3点）

### 変更前 → 変更後

| 機能 | 現行Free | 推奨Free | Pro |
|---|---|---|---|
| **スケジュール** | 毎日00:00-23:59固定 | **曜日選択のみ可** (時間帯はPro) | 曜日x時間帯 自由設定 |
| **ブロックサイト** | プリセット最大3件 | **プリセット最大5件** | 無制限+カスタムドメイン |
| **運動統計** | なし | **直近7日間** | 全期間+週次/月次レポート |
| ロック時間 | 5/10/20分の3択 | 5/15/30分の3択 | 1分〜自由設定 |
| 運動回数 | 5〜20回 | 5〜30回 | 制限なし |
| カスタムドメイン | 不可 | 不可 | 無制限 |
| アダルトブロック | 利用可能 | 利用可能 | 利用可能 |

### なぜこの境界がCVRを最大化するか

1. **スケジュール曜日選択をFreeに**: 現行の24/7固定はアンインストールの最大要因。「平日だけ」を許可することで離脱を防ぎ、「時間帯も変えたい」でPro動機が生まれる
2. **サイト5件に拡大**: 3件はYouTube+X+Instagramで埋まり余裕がない。5件で「使える」体験を提供し、6件目でUpgrade Moment
3. **7日間統計をFreeに**: THE TOLL最大の差別化「運動習慣化」を数値で実感させ、「もっと見たい→Pro」の動機を作る

## 3. 推奨トライアル設計: Reverse Trial (14日間)

```
Day 1-14:   全機能フル開放 (Full Pro)
Day 15-21:  カスタムドメイン不可、スケジュール制限
Day 22-28:  ブロックサイト5件まで、運動回数制限復活
Day 29+:    完全Free制限
```

**現行7日→14日に延長する理由**:
- 7日間では習慣化前にトライアル終了（習慣定着には平均21-66日）
- 14日間で週末を2回経験でき、価値を十分に体感
- 段階的制限（Reverse Trial）で損失回避の心理が働き、CVR +30-50%見込み

**Day 12通知例**: 「トライアル終了まで3日。あなたは累計487回のスクワットを達成！この記録を守るには→」

## 6. 技術実装の優先順位

### Step 1: セキュリティ最優先 (1-2日) ← リリースブロッカー

- `unlock_session` にPro強制チェック追加（現在フリーでもサーバー側解除可能）
- `device_links` RLS強化（現在 `using(true)` で全件公開）

### Step 2: Feature Flag基盤 (1日)

- `plan_limits` テーブル作成 → プラン境界をDB管理化
- `get_plan_limits` RPC → クライアントが動的に制限値取得
- ハードコード制限値をDB駆動に置換 → A/Bテスト対応

### Step 3: グレースフルデグラデーション (1日)

- Pro→Free降格時: データ消去なし、`_inactive`キーに退避 → 再昇格で復元
- ユーザー通知: 「サイト◯件が一時停止中。Upgradeで復元→」

### Step 4: トラッキング基盤 (2-3週間)

- `user_events` テーブル + `track-event` Edge Function
- Chrome拡張/PWAにイベント埋め込み
- ファネルダッシュボード構築

## 11. 現行設計への適用判断（採用/保留）

現行実装（2026-02-15時点）に対して、5レポートの提案を以下の方針で適用する。

| レポート | 評価 | 適用方針 |
|---|---|---|
| 01 Product Strategy | 高 | Free/Pro境界の考え方を採用。特に「Pro=パーソナライズ価値」を中核に据える。 |
| 02 Data Analysis | 高 | 最小イベントから段階導入。まず課金導線・ロック解除導線の計測を優先。 |
| 03 UX Growth | 中 | 導線改善とコピーは採用。価格UI案は現行フローに合わせて再編集して適用。 |
| 04 Billing Architecture | 最重要 | Server-side enforcementを最優先で実装（`unlock_session` ガード、RLS強化、plan_limits化）。 |
| 05 Pricing Strategy | 中〜高 | 価格・トライアル案は仮説として採用し、A/Bテストで検証して確定。 |

### 実行優先順位（確定）
1. 課金・権限制御のサーバー側強化（Report 04）
2. トラッキング最小実装（Report 02）
3. Free/Pro境界と導線チューニング（Report 01 + 03）
4. 価格・トライアル最適化の実験運用（Report 05）

### 当面の意思決定
- 「全部を一気に採用」はしない。実装コストとリスクの低い順に適用する。
- リリース品質に直結する項目（権限制御・課金整合）を先に固定し、CVR改善は計測基盤の上で回す。
