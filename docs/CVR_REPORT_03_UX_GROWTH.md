## 1. ペイウォール画面のUI改善案

### 1.1 現状の課題

現在の `pricing.html` は以下の2ボタン構成のみで、CVR上の問題が多い。

```
[MONTHLY PLAN]
[YEARLY PLAN (Recommended)]
```

**問題点:**
- 価格が表示されていない（「checkout で確認」と書かれており不安感を与える）
- Freeとの機能差が不明瞭 → 「なぜ払うのか」が伝わらない
- Yearlyが視覚的に目立っていない（ボタン色の差のみ）
- 社会的証明がゼロ
- スクロールなしで全情報が見えるが、逆に「情報が少なすぎて信頼できない」印象

### 1.4 Free vs Pro 機能比較表

| 機能 | Free | Pro |
|------|------|-----|
| ブロックサイト数 | 3件まで | 無制限 |

| ロック猶予時間 | 5/10/20分のみ | 1〜120分・自由設定 |

**UI実装案:**

### 1.5 CTA最適化

**Yearlyを「推奨」として強調する設計:**

```css
/* Yearlyボタンは枠線+背景+バッジで目立たせる */
.btn-yearly {
  background: #c6ff00;  /* ブランドカラー（volt yellow） */
  color: #000;
  position: relative;
}

.btn-yearly::before {
  content: "BEST VALUE";
  /* バッジとして右上に配置 */
}
```

**CTA文言:**
- Yearly: 「今すぐ始める — 2ヶ月分無料」
- Monthly: 「まず月払いで試す」

**アンカリング効果の活用:**
Monthlyの価格を先に目立たせてから、Yearlyの割引率を見せる
→「月1,800円」→「Yearlyなら月1,200円（34%お得）」

## 2. アップグレード導線の最適化

### 2.1 ペイウォール遭遇タイミングの設計

ユーザーが「この機能を使いたい」と思った瞬間に訴求することが最もCVRが高い。
遭遇させるタイミングを「欲求が高まった瞬間」に合わせて設計する。

#### タイミングA: カスタムドメイン追加時（最重要）

**現在の動作:** `ADD`ボタン押下 → グレーアウト表示 → 何も起きない

**改善案:**

```
ユーザーがカスタムドメインを入力してADDをクリック
         ↓
ポップアップorインラインメッセージ表示:
┌─────────────────────────────────┐
│ 🔒 PRO FEATURE                  │
│                                  │
│ カスタムドメインの追加はProプラン  │
│ で利用できます。                  │
│                                  │
│ [UPGRADE TO PRO →]  [キャンセル] │
└─────────────────────────────────┘
```

- ユーザーが「やりたかったこと」の直後に表示するため、動機が最高潮
- 「無視して続ける」オプションは提供せず、UPGRADEかキャンセルのみ

#### タイミングB: ロック時間を1分/カスタム設定しようとしたとき

**現在:** 1分ラジオボタンは非表示or disabled

**改善案:**
- ラジオカードに薄くPROバッジを表示
- クリックするとインラインで「Proプランで利用可能」メッセージ + UPGRADE導線

#### タイミングC: ブロックサイト4件目を追加しようとしたとき

```
インラインメッセージ:
"Freeプランはサイト3件まで。
 Proにアップグレードして無制限に追加できます。 → [UPGRADE]"
```

#### タイミングD: スケジュール曜日・時間を詳細設定しようとしたとき

- Freeでは曜日チェックボックスをdisabledに
- 「詳細スケジュールはPro機能」のバナーを設定グループ内に常時表示

### 2.2 コンテキスト別アプローチ

#### ロック画面から (content.js / overlay.css)

**現在:** ロック画面はセッションID + QRコードのみ

**改善案 — ロック画面下部に控えめなPROプロモーション帯:**

```
┌─────────────────────────────────┐
│  [メインロック画面]               │
│  SESSION ID: XXXX-XXXX          │
│  [QRコード]                      │
│  SCAN TO START EXERCISE         │
│                                  │
│  ─────────────────────────────  │
│  ⚡ PRO TIP: Set 1-min grace    │
│  period for quick returns       │
│  → Upgrade to Pro              │  ← 小さく・控えめに
└─────────────────────────────────┘
```

- ロック中は「運動に集中させる」が主目的のため、PRO訴求は控えめ
- 「PRO TIPとして有用な情報を提供しつつ」升格を促す

#### 設定画面から (popup.html / popup.js)

**現在:** FREEユーザーはPROバッジが表示されるのみ

**改善案 — 設定画面トップにプラン状態バナー:**

```
FREE プラン使用中
あと◯日でキャンペーン終了。Proにアップグレードして全機能を解放 → [UPGRADE]
```

- トライアル期間中は「残◯日」カウントダウン表示
- FREEユーザーには「今日だけ◯%OFF」などの限定性訴求（実施する場合）

#### トライアル終了時

**タイミング:** `trial_ends_at` を過ぎてFREEに戻った最初の起動時

**改善案 — 専用モーダル（popup.js で1回だけ表示）:**

```javascript
// トライアル終了後初回起動時のみ表示フラグを使う
if (planStatus === 'free' && wasOnTrial && !trialEndedModalShown) {
  showTrialEndedModal();
  localStorage.setItem('trial_ended_modal_shown', 'true');
}
```

**モーダルコンテンツ:**
```

│  7日間のトライアルが終了しました  │
│                                  │
│  カスタムドメイン・詳細スケジュール│
│  などのPro機能が制限されています。 │
│                                  │
│  [Proを続ける — ¥1,200/月〜]    │
│  [Freeで続ける]                   │
└─────────────────────────────────┘
```

### 2.3 ナッジの設計

**ナッジの原則:** ユーザーを「騙す」のではなく「気づかせる」こと

| ナッジ | 実装場所 | 頻度 |
|--------|----------|------|
| Proバッジ（機能名横） | popup.html 各設定グループ | 常時 |
| インラインアップグレード提案 | Pro機能クリック時 | その都度 |
| プラン状態バナー（残日数） | popup.html 上部 | トライアル中は常時 |
| トライアル終了モーダル | popup.js | 1回のみ |
| ロック画面PRO TIP | content.js overlay | 常時（控えめ） |

**やってはいけないこと:**
- ダークパターン（「Proを続ける」だけ目立たせて「Freeに戻る」を見つけにくくする）
- 毎回ポップアップで訴求する（ユーザーが拡張をOFFにする原因になる）
- 価格を隠す（「checkout で確認」は不信感を生む）

## 3. フリクション分析と改善

### 3.1 現在のFree→Pro導線における離脱ポイント

```
[UPGRADEボタン] → pricing.htmlポップアップ → Stripe Checkout → 完了
```

#### 離脱ポイント①: UPGRADEボタン → pricing.html

**問題:** pricing.htmlを「新しいウィンドウで開く」実装
- ポップアップブロックにかかる場合がある
- Chromeポップアップのタイトルバーが「ちゃんとしたページ感」を出せない

**改善案:**
- Chromeポップアップ内でインライン表示（popup.htmlに pricing セクションを追加）
- または `chrome.tabs.create()` でタブで開く（より信頼感がある）

#### 離脱ポイント②: pricing.html → Stripe Checkout

**問題:** 「Final amount shown at checkout」という文言が不安感を与える
- ユーザーは「いくら取られるかわからないまま次へ進む」ことを避ける

**改善案:**
- pricing.htmlに実際の価格を明示する
- 「安全なStripe決済」のロゴ/バッジを追加
- 「いつでもキャンセル可能」の一言を添える

#### 離脱ポイント③: billing-return.html

**問題:** 決済完了後のフィードバックが最小限
- `BILLING RESULT` というタイトル + メッセージ + CLOSEボタンのみ
- ユーザーに「次に何をすればいいか」が伝わらない

**改善案:** 後述の「3.3 決済完了後の成功体験設計」参照

---

### 3.2 チェックアウトフローの簡素化案

**現在のステップ数:** UPGRADEボタン(1) → pricing.html(2) → Plan選択(3) → Stripe Checkout(4) → billing-return.html(5) = **5ステップ**

**削減案:**

**オプションA: プラン選択をポップアップ内に埋め込む（推奨）**

```
popup.html の フッター部分に:
[MONTHLY ¥1,800] [YEARLY ¥14,400 ★]  ← UPGRADEボタンの代わりに2択を直接表示
```
→ pricing.htmlを経由せずに直接Stripe Checkoutへ
→ **5ステップ → 3ステップ**

**オプションB: 最もお得なYearlyへ直接誘導**

```
[UPGRADE TO PRO — ¥1,200/月] というボタン1つ
(クリックするとYearly Checkoutへ直行)
+ 「月払いはこちら」のテキストリンク
```
→ 心理的な「選択の負荷」を排除

**オプションC: 既存の2ボタンはそのままで、ページのUIを改善（最小コスト）**

- pricing.htmlの改修のみで対応
- ステップ数は変わらないが、各ページでの離脱率を下げる

## 4. 「Aha Moment」の設計

### 4.1 Freeユーザーが「Proにする価値がある」と感じる瞬間の特定

THE TOLLのコアバリューは **「意志力に頼らず、仕組みで行動を変える」** こと。

| Aha Momentの候補 | 発生する状況 | CVRへの影響度 |
|-----------------|-------------|--------------|
| A: カスタムドメインを追加したくなる | 仕事の集中を妨げる特定サイト(Slack, Notion等)を追加しようとする | ★★★★★ |
| B: ロック時間を細かく設定したくなる | 「5分じゃ短い、7分にしたい」と感じる | ★★★★ |
| C: スケジュールを週末だけにしたくなる | 土日の過ごし方を変えようとする | ★★★ |
| D: 運動が習慣化して「もっと追い込みたい」 | 20回スクワット以上に挑戦したくなる | ★★★ |
| E: 7日トライアルで全機能を使って「手放せない」 | トライアル終了後にFreeで不便を感じる | ★★★★★ |

**最重要のAha Moment: A（カスタムドメイン追加時）とE（トライアル終了後）**

---

### 4.2 その瞬間を意図的に作り出す仕掛け

#### 仕掛け①: トライアル開始の積極的な促進

**現在:** トライアルの開始条件が不明確（おそらく新規登録時に自動付与）

**改善案:**
- 初回ログイン後のポップアップで「**7日間の無料トライアルを開始**」を明示
- 「今すぐ全機能を無料で試す」CTAを目立たせる
- トライアル中は拡張バッジに「TRIAL: 残6日」を表示（毎日意識させる）

#### 仕掛け②: カスタムドメインのプレビュー体験

**改善案:**
- Freeユーザーがカスタムドメイン入力欄に入力すると、
  「このサイトをブロックするとどうなるか」をプレビューとして見せる
- 「追加するにはProにアップグレード」でCTA表示

```javascript
// カスタムドメイン入力時にリアルタイムプレビュー
customDomainInput.addEventListener('input', (e) => {
  if (isPro) return;
  if (e.target.value.length > 3) {
    showProPreviewHint(e.target.value); // 「このサイトをブロック可能 → Pro機能」
  }
});
```

#### 仕掛け③: 運動習慣ストリーク表示

**改善案:**
- スマホアプリのホーム画面にストリーク（連続日数）を表示
- 「◯日連続で運動しています！」をモチベーションに変換
- **「Proにするとストリークデータの詳細が見られる」** というPro価値を追加

#### 仕掛け④: Proユーザーの「生産性レポート」（将来機能）

- 「今月◯時間、SNSへの衝動を運動に変換しました」という週次レポート
- Freeユーザーには「Proでより詳しい分析が見られます」と訴求

---

### 4.3 Pro機能のプレビュー・体験設計

**プレビューのデザイン原則:**
- 「見えるけど触れない」ではなく「触れる + 即座にアップグレード促進」
- フラストレーションを最小化しながら欲求を高める

**実装パターン:**

```
パターン1: グレーアウト + クリックで説明モーダル（現在の実装）
  → 改善: モーダルに価格・機能比較・CTA を追加

パターン2: インライン試用（1回だけ使える）
  例: カスタムドメインを1件だけ試せる
  「あと2件追加するにはProへ」
  → ユーザーが価値を体験してから購入

パターン3: 設定のプレビュー（保存できない）
  例: スケジュールを自由に設定できるUIを触れる
  保存ボタン押下時に「Proプランで保存可能」と表示
```
